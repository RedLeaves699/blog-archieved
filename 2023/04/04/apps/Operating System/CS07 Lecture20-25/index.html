    内存使用与分段

    实现内存的使用

    最简单的让程序执行的办法，就是把程序直接加载到内存中。为 PC 指定一个首地址。然后 CPU 就可以开始执行了。简而言之，程序在定位后，我们就能使用内存了。

    如何定位程序：但程序的第一句并不一定会加载到 0 处，所以我们还要想办法处理程序内部的跳转，这就需要我们指定程序的起始地址（又称基地址） base，每次执行跳转语句时，要进行地址翻译，即计算跳转地址 base + offset（程序内相对地址）的值。

    程序的位置并非一成不变：程序在内存较小的情况下执行时，系统可能会换出中断时间较长的程序，将新的程序换入内存。所以，程序基地址应该保存在PCB中。

    如何重定位程序：程序重新被换入内存时要进行载入时重定位。首先，操作系统在内存中找到空闲内存，将内存基地址赋给 PCB 的 base。然后把程序写入这段内存，最后，每次执行执行指令时使用 base + offset 计算。部分嵌入式系统的采用编译时重定位，即将一段程序写在内存绝对地址。

    分段方法的引入

    一个程序有很多部分组成：主程序，变量集，函数库，动态数组，堆栈。这些段中，每个段有各自的特点，用途，如代码段是只读的，而且不能动态增长。我们载入一个程序时，不需要把整个程序的所有内容都加载到一起，而是分段加载，这使得用户可以独立的考虑每个段，实现分治的思想。对内存使用也更高效。

    程序的分段：不是将整个程序，而是将各段分别存入内存，段的长度是可变的。

    因为一个程序被分为了多个段，所以一个进程的 PCB 要放多个段基址，这些段基址组成了进程的段表（LDT）。把操作系统看作一个进程，GDT 就是操作系统的段表。

    进程的段表主要信息如下：

    段号 基址 长度 保护 其他属性
    0 180K 150K W ...
    1 360K 60K R/W ...
    2 70K 110K R/W ...
    3 460K 40K R ...

    段的全部属性由段描述符描述：

    在进程执行时，操作系统定位段内数据的完整过程如下： 1. 从 gdtr 寄存器中获得 GDT 基址。 2. 从LDTR寄存器中获取LDT所在段的位置索引(LDTR高13位)。 3. ldtr 寄存器存放着 LDT 所在段的位置索引。 4. 段基址 + 段内偏移得到内存线性地址，当前程序 CS 寄存器的值为0，所以进入基址 180K 的这个段寻址。

    总结：程序放入内存中的步骤

    1. 将程序分段（代码段，数据段，堆栈等），主要由编译器完成
    2. 在内存中划分空闲区域，载入程序（操作系统内存管理）
    3. 建立映射表，程序和PCB关联（操作系统进程管理）

    内存的分区和分页

    内存如何分割

    Q1 : 操作系统需要为不同长度的段分配内存。应该如何分配？ A1 : 分配方法没有对错，只有优缺点。目前有两种思路，一种是最优适配，分配可用空间中最接近当前请求内存大小的一块内存。一种是最差适配，与最优适配正好相反，分配可用空间中最大的一块内存。

    Q2 : 操作系统的段内存请求不规则，此时使用哪种方法最好？操作系统是如何实现的？ A2 : 最佳适配。浪费内存比最差适配少一些。操作系统使用分页方法来分割物理内存，不用分段的方法。

    分段方法的缺点

    操作系统不断为段分配空间，大大小小的段之间会产生很多小内存碎片。导致总空闲空间大于请求的空间却无法分配内存。

    可以让程序在内存中移动，使得它们紧凑的排列，即内存紧缩的方法改善内存碎片问题。然而这种方法需要逐单元移动内存，开销太大。

    分页方法的引入

    分段的碎片好像切面包的碎屑，为了避免这种碎屑，我们把面包切成均匀的片。在操作系统实现上，就是把内存分成大小相同的页。页大小按惯例来说是4K。分页之后，程序的存储方式如表所示。由图中我们可以发现，一个程序最大的内存损失只有 4K - 1，远远小于段式内存管理，避免了碎片的问题。

    分段和分页

    用户执行程序希望程序分段，分段的代码和数据便于管理，程序寻址开销更小。物理内存希望分页，这样可以减少浪费。

    多级页表和快表

    看起来，页表越小，内存空间的浪费越少。但我们还应该关心页表的大小。

    算一下页表的大小：一个页表可涵盖 4k 个地址，页表的页框连续存储，所以有\(2^{32} / 4\text{K} = 1\text{M}\)个页框，一个页框号占用 4B，所以一个程序的页表大小为 4MB。如果开 100 个进程，页表占用的内存空间就高达 400MB。损失太大了。

    32位系统的寻址空间为 0-4G。但实际上，一个进程运行时，大部分逻辑地址根本不会用到。4M的页表有很多项是空的，这些空表项占用了很多空间。

    方法一：只存放用到的页

    只为用到的逻辑页分配页表项。

    页号 页框号
    0 5
    1 1
    3 6

    如果每页中的页号不连续，就需要比较，查找，最好的方式是二分查找，复杂度\(O(\log n)\)。\(\log 2^{20}=20\)，开销太大。远不如顺序索引，所以我们保证页表表项的连续。

    为页表创建一个页目录。页目录可以决定页表的存储位置，当然也可以决定是否为页表分配存储空间。所以多级页表可以节省空间。多级页表结构如图所示：

    优化多级页表的时间效率——TLB快表

    增加一级页表后，要想访问一个物理地址，就要先去内存的页目录中查页表号，再从页表中中查页框号，最后再用页框号加上偏移找到物理地址，共访存3次，开销很大。

    如果创建一个表，能够直接记住当前页号对应的页框号，就不必在多级页表查找上浪费时间了。这样的表一般是硬件实现的，成为快表（TLB）。TLB是特殊的 cache 存储器，一般采用全相联或组相联实现。

    Linux0.11段页式内存管理的实现

    从fork中的内存分配开始。。。

    Q1 : 段页式内存下程序如何载入内存？ A1 : 整个过程分五步： 1. 为程序分割虚拟内存的空余空间 2. 将用户的段和虚拟内存区域关联 3. 在物理内存寻找空闲页(get_free_page) 4. 建立段表和页表 5. 使用重定位使用内存（验证）

    前面讲过sys_fork()中调用的copy_process函数，这个函数关于内存的操作被略过了，现在重新审视关于内存的操作。

    int copy_process(int nr, long ebp, ...) {
    ...
    copy_mem(nr, p);
    ...
}

int copy_mem(int nr, task_struct *p) {
    unsigned long new_data_base;
    // 64M * nr，指示分割虚拟内存的大小
    new_data_base = nr * 0x4000000;
    // 填写段表
    set_base(p->ldt[1], new_data_base);
    set_base(p->ldt[2], new_data_base);
    unsigned long old_data_base;
    old_data_base = get_base(current->ldt[2]);
    // 和父进程共用一个页框，拷贝父进程页表即可
    copy_page_tables(old_data_base, new_data_base, data_limit);
    ...
}

void copy_page_tables(unsigned long from, unsigned long  to, long size) {
    unsigned long * from_page_table;
    unsigned long * to_page_table;
    unsigned long this_page;
    unsigned long * from_dir, * to_dir;
    unsigned long nr;
    // 右移22位找到页目录项编号，再乘4找到这个目录号虚拟内存起始地址
    from_dir = (unsigned long *) ((from >> 20) & 0xffc);
    to_dir = (unsigned long *) ((to >> 20) & 0xffc);
    // size : 应该复制多少个页表
    // 把不足一个4MB(一个页表所能控制的内存长度）的size取整为一个4MB
    size = (unsigned long *) (size + 0x3fffff) >> 22;
    for (;size-->0;from_dir++, to_dir++) {// 页目录
        from_page_table = (0xfffff000) & *from_dir;
        to_page_table = get_free_page(); // 多级目录需要做出新页表
        // 下面这句设置新页表的属性信息，(usr, r/w, present)
        *to_dir = ((unsigned long) to_page_table) | 7; 
        for (;nr-->0;from_page_table++, to_page_table++) {// nr=1024
            this_page = *from_page_table;
            this_page&=~2; //只读
            *to_page_table=this_page;
            this_page -= LOW_MEM;//读写控制
            this_page >>= 12;
            mem_map[this_page]++;
        }
    }
}

    Linux0.11页式管理的特殊性

    操作系统0号进程的作用有两个，一个就是创建1号进程，另外就是当没有就绪的进程时，就会执行 cpu_idle 函数，让 cpu 处于空转状态。

    在虚拟地址空间中，操作系统进程0的 GDT 与 LDT 重合，占用从0开始共64M的地址，进程1是所有用户态程序的父进程，占用从64M开始共64M的地址。然后就是进程2，进程2的代码段LDT[1]和数据段LDT[2]共用128M开始共64M的地址。每个进程的虚拟地址互不重叠。

    进程虚拟地址不重叠是对内存管理的一种简化，意味着我们对所有进程，只需要使用一个统一的页表就可以完成地址转换。在高级Linux系统中，页号很有可能重叠（进程调度和内存换入和换出就可能导致这个问题），所以每个进程应该有自己的页表。

    内存的换入换出

    为什么使用虚拟内存？

    为了使用段页式内存管理，用虚拟内存作为段式管理和页式管理的桥梁。内存的换入换出是实现虚拟内存的最重要的技术。

    虚拟内存可以用换入换出实现大内存。即使虚拟内存大于物理内存，计算机仍能正常工作。用户能见到的内存是 4G 的虚拟内存，当用户使用虚拟内存 0G ~ 1G 的区域时，系统从磁盘向物理内存装载相应数据，建立映射表，当用户使用虚拟内存 3G ~ 4G 的区域时，系统换出当前数据到磁盘上，再从磁盘向内存装载数据，建立新的映射表，这样就可以实现4G，甚至“无限大”的虚拟内存。

    内存的换出策略

    内存空间是有限的，如果内存被占满，再调入新的页时，就要淘汰内存中一个旧的页到磁盘上。该淘汰谁呢？淘汰的过程是什么样的？

    现在有3个页框，页面引用序列为ABCABDADBCB FIFO：先入先出

    A B C -> D B C -> D A C -> D A B -> C A B
    • 评价算法好坏：缺页次数，FIFO为7

    MIN：选最远将使用的页淘汰，是最优方案:

    A B C -> A B D -> C B D
    • 缺页次数：5。实际中不能用，因为要准确的预测未来。

    LRU：选最近最长一段时间没有使用的页淘汰（最近最少使用）

    A B C -> A B D -> C B D
    • 缺页次数：5。实际表现效果较好，充分利用了程序局部性，被广泛使用。

    LRU的实现：

    直观想法：时间戳（Time stamp），选择时间戳最小的页。然而，如果每次都对页表项中的每一个页进行修改时间戳的操作，访存的代价是巨大的，时间戳也可能溢出。所以这种方法不可行。

    想法2：定长队列，队首淘汰，每次地址访问都需要修改队列（修改10次左右），实现代价大。

    虽然 LRU 方法表现优秀，但放在实际系统，很难高效实践。所以操作系统中往往使用 LRU 近似算法。

    LRU 近似实现（Second Chance Replacement算法）： 用最近没有使用近似最近最少使用 1. 每个页增加一个引用位 R，每次访问一页时，硬件自动设置该位。 2. 选择淘汰页：扫描该位，是1时清0，并继续扫描；是0是淘汰该页。

    改进 SCR —— Clock 算法：

    刚刚的 Clock 算法在缺页很少的时候，程序不停执行，所有的页引用位 R 都会被置为1，发生缺页时，指针转一圈，最后才能把刚指针开始指向的页置换，指针转一圈降低了效率。而且，Clock 算法退化成了 FIFO 算法，失去了 LRU 的意义。

    发生上述情况的原因？——记录了太长的历史信息。所以我们可以再用一个扫描指针定时清除 R 位。清除 R 位的指针移动速度快，用来选择淘汰页的指针移动速度慢。

    给一个进程分配多少页框

    分配太多，请求调页导致的内存高效利用就没用了。 分配太少，会发生颠簸（thrashing）。

    颠簸（thrashing）：系统内进程增多 \(\Rightarrow\) 每个进程的缺页率增大 \(\Rightarrow\) 缺页率增大到一定程度，进程总等待调页完成 \(\Rightarrow\)CPU利用率降低 \(\Rightarrow\) 进程进一步增多，缺页率更大 \(\Rightarrow\) ...
