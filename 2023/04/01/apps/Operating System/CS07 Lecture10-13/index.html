    用户级线程

    Q1 : 线程是什么？和进程有什么区别？ A1 : 进程包括其占用的资源和指令序列，因为进程拥有独立的资源，所以切换时要恢复资源和改变PC的值。多个线程之间共享资源，在线程切换时，只需要改变 PC 的值，而不需要恢复线程的资源。线程保留了并发的优点，避免了进程切换复制资源的代价。

    Q2 : 线程的实用性如何体现？ A2 : 以一个网页浏览器渲染网页为例： 一个线程用来从服务器接收数据，一个线程用来处理图片(如解压缩)，一个线程用来显示文本，一个线程用来显示图片。四个线程之间要共享资源，同时显示在屏幕上。如果用四个进程，那么在最后还需要进程之间的相互通信才能将资源汇总到一起。 浏览器的代码大致如下：

    void WebExplorer() {
    char URL[] = "http://redleaves699.github.io";
    char buffer[1000];
    pthread_create(..., GetData, URL, buffer);
    pthread_create(..., Show, buffer); 
}
void GetData(char *URL, char *p){...};
void Show(char *p){...};

    如何完成线程切换？我们可以模拟一下：

    在此之前，我们实现完成线程切换Yield()函数：

    void Yield() {
    find address
    jmp_to ...
}

    按照C语言的规则，我们执行线程1的函数A，函数 A 在地址100调用了函数 B，程序将下一条指令的地址104压栈，进入函数 B，函数 B 在地址200处调用函数Yield()，将地址204压栈，此时跳转到线程2。线程2如线程1一样，执行完Yield()函数，此时栈的状态为： 404 \(\leftarrow\) esp 304 204 104 现在应该执行地址204处的指令。正常执行后，函数B()执行完了，将要执行汇编指令ret。ret会将栈顶元素的值赋给PC并弹出栈顶元素。我们希望线程1中的函数 B 在执行完后返回线程一的 A，但此时出现了问题，PC 回到了地址404。

    所以线程之间不能共享栈资源。线程的栈是独立的。所以我们在线程切换时不能只切换PC的值，也要切换栈。所以应修改Yield()函数的实现，其中TCB是一种和PCB比较类似的结构，用来记录线程信息：

    // Pseudo code
void Yield(thread a, thread b) { // thread a -> thread b
    TCB(a).esp = current_esp
    current_esp = TCB(b).esp
    jmp_to address
}

    修改后，程序执行到204指令的栈状态为：

    栈thread1 esp 栈thread2 esp
    104 304
    204 \(\leftarrow\) esp 404

    在执行完 B 后，函数执行ret指令，但此时thread1.esp == current_esp == 204，所以PC又会回到204了，不太对。

    想想这样的情况是如何产生的。

    204是 B 执行完Yield()函数之后被压栈的。所以，thread1栈内保存的地址就是 B 中Yield()函数的下一条指令。

    我们在Yield()函数直接切换栈，利用 D 中Yield()函数的ret指令即可实现弹出204地址并执行的效果。此外，为了让程序在第一次调用Yield()函数时找到执行地址，我们初始化栈时，要压入程序首地址。

    // Pseudo code
void Yield(thread a, thread b) { // thread a -> thread b
    TCB(a).esp = current_esp
    current_esp = TCB(b).esp
}

    应用这样的思想，我们可以构建相关的数据结构了：

    void ThreadCreat(A) {
    TCB *tcb = malloc(); // 创建TCB
    *stack = malloc(); // 创建栈空间
    *stack = A; // 100 : 在栈中压入程序待执行地址
    tcb.esp = stack // esp = stack的首地址
    tcb.状态 = 就绪;
    tcb入队;
}

    当然，真正的yield函数还要具有调度功能：

    void Yield() {
    保存现场;
    进程调度;
    恢复现场;
} // ret,切换线程

    Q3 : 为什么说是用户级线程？ A3 : yield()是用户程序。

    Q4 : 用户级线程有什么缺点？和内核级进程有什么区别？ A4 : 一个进程未响应，则这个进程的线程均未响应。具体图示如下：

    内核级线程

    内核级线程是多核系统必须支持的功能。多个内核级线程可以让多个 CPU 核并行工作。因为要进入内核中执行程序，所以内核级线程需要同时使用用户和内核的资源。一个用户级线程需要使用一个用户栈，而一个内核级线程就需要同时使用用户栈和内核栈，即一套栈。同样，一个用户级线程的TCB需要关联一个用户栈，一个内核级进程的TCB就要关联一套栈。在线程调度时，内核栈和用户栈要同时切换。

    内核级线程的切换——内核栈KS切换法

    下面用一个示例演示内核栈切换方法。

    程序现在的状态是内核态，现在执行sys_read()系统调用。这个系统调用会启动调度函数并切换当前线程。切换线程时，操作系统把当前线程的一套栈全部保存在TCB中，然后，使用下一个线程的TCB恢复现场。在切换后，switch_to()函数执行ret指令将PC改变为下一个线程的内核程序地址。由于程序代码在用户态中执行，所以最后再用 CS 和 PC 切回用户程序，用 SS 和 SP 恢复用户栈。线程调度至此完成。

    内核级线程的切换——TSS切换法

    在 Linux0.11 中，实现进程切换的方法与上面的方式有一点差别，它利用了TSS（Task Struct Segment）方法。

    TSS 是一个段，用于存储程序运行时的所有寄存器内容，相当于为程序拍摄了一张快照。TSS 描述符存在 GDT 表中，是指向 TSS 的指针。访问 GDT 中的描述符需要使用选择子 TR。

    使用 TSS 方法时，需要用ljmp指令，ljmp指令将当前现场保存到当前 TSS 块中，并为 TR 赋值，最后，TSS 的值就会自动载入到寄存器中。

    TSS切换代码

    /*
 * switch_to(n) should switch tasks to task nr n, first
 * checking that n isn't the current task, in which case it does nothing.
 * This also clears the TS-flag if the task we switched to has used
 * tha math co-processor latest.
 */
#define switch_to(n) {\
struct {long a,b;} __tmp; \
__asm__("cmpl %%ecx,_current\n\t" \
    "je 1f\n\t" \
    "movw %%dx,%1\n\t" \
    "xchgl %%ecx,_current\n\t" \
    "ljmp %0\n\t" \
    "cmpl %%ecx,_last_task_used_math\n\t" \
    "jne 1f\n\t" \
    "clts\n" \
    "1:" \
    ::"m" (*&__tmp.a),"m" (*&__tmp.b), \
    "d" (_TSS(n)),"c" ((long) task[n])); \
}

    实现内核级线程切换的数据结构

    为了实现上述方法所构建的数据结构，比起用户级线程，只是多了一个内核栈而已。

    // pseudo code
void ThreadCreate(...) {
    TCB tcb = get_free_page();
    *krlstack = ...; // 内核栈
    *userstack传入;
    填写两个stack; // 用TSS可以不填
    tcb.esp = krlstack;
    tcb.状态 = 就绪;
    tcb入队;
}

    总结：线程，进程切换五段论

    1. 中断入口
    2. 中断处理（IO中断，时钟中断等）
    3. 任务调度算法
    4. 内核栈切换
    5. 内核向用户第二级切换

    总结：用户级线程、内核级线程的对比

    用户级线程 核心级线程 用户+核心级
    利用多核 差 好 好
    并发度 低 高 高
    代价 小 大 中
    内核改动 无 大 大
    用户灵活性 大 小 大

    Linux0.11内核级线程实现

    创建相同进程的框架代码（Linux0.11不支持线程）：

    int main() {
    A();
    B();
}

void A() {
    fork(); // 系统调用，会引起中断
}

fork:
    mov %eax, __NR_fork
    INT 0x80 // cpu找到当前内核栈，压入SS:SP，EFLAGS，返回地址=PC+4，中断处理，调用system_call
    mov res, %eax
    返回res // 源代码这里是内嵌汇编返回res

_system_call:
    push %ds ... %fs
    pushl %edx ...
    call sys_fork
    pushl %eax
    movl _current, %eax // _current:当前程序PCB
    // state + %eax,为0就绪，非0阻塞，state(%eax) 在sys_fork()函数中被置为非0
    cmpl $0, state(%eax)
    jne reschedule // 上面讲的切换是这里引发的
    cmpl $0, counter(%eax) // 看时间片是否为0
    je reschedule
    ret_from_sys_call:

    执行完sys_fork()以后，子进程会继续运行父进程的代码。

    reschedule:
    pushl $ret_from_sys_call
    jmp _schedule // schedule是C函数

ret_from_sys_call:
    popl %eax // sys_fork的返回值
    popl %ebx ...
    pop %fs ...
    iret // 返回到int 0x80后面执行

    这是创建新线程的主程序，从_sys_fork开始CreateThread：

    _sys_fork:
    push %gs; pushl %esi
    ...
    pushl %eax
    call _copy_process // 这是一个C函数，输入参数在内核栈中
    addl $20, %esp
    ret

    // 参数从栈中自上而下逐个读取，是之前汇编压进来的
    ...
    // 申请内存空间
    p = (struct task_struct *) get_free_page();
    // 创建内核栈
    p->tss.esp0 = PAGE_SIZE + (long) p;
    p->tss.ss0 = 0x10;

    p->tss.esp0 = ss & 0xffff;
    p->tss.ss0 = esp;
    // 填写两个栈...，用TSS不需要做。。。

}

    创建自己之外的进程：

    方法：从exec系统调用iret出来的时候，将原来子程序的返回地址改成将要执行函数的入口地址。

    过程：子进程fork()返回后，即执行exec(cmd)程序，该程序会使用系统中断，所以会执行INT 0x80进入内核态，进入内核态后调用_system_call，压入必要参数后调用sys_execve。之后，sys_execve向栈中压入 EIP(0x1c) + %esp，目的是给C语言函数do_execve()传送参数。这个参数表示的位置如图红色字体所示。接着，将eip[0]，即ret地址，修改为函数入口，eip[3]，即SP地址，修改为函数用户栈地址。

    之后，将参数逐个出栈，sys_execve执行ret指令就会直接跳转到待执行程序的入口处，对exec()的成功调用永远不会返回。当子进程结束（exec()想执行的程序执行完，然后到wait(0)），将资源还给父进程，然后父进程执行循环，等待输入。

    int main() {
    while (1) {
        scanf("%d", cmd);
        if (!fork()){exec(cmd);}
        /* 先调用fork创建子进程，父进程和子进程执行同样的代码，父进程fork()不会返回1，
        只有子进程fork()返回0，此时子进程执行exec()系统调用。*/
        /* 父进程会在此阻塞，等待回收子进程资源，wait一次只能回收一个子进程，回收成功会返回-1 */
        wait(0);
    }
}
