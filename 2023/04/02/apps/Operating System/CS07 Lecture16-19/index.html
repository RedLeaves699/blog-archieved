    进程同步与信号量

    Q1 : 进程怎么同步？ A1 : 让进程走走停停来保证多进程合作的合理有序。没有资源就停，有资源就走。

    生产者-消费者实例

    生产者生产 item 占用缓冲区资源，消费者使用 item 释放缓冲区资源

    生产者

    while (1) {
    while(counter== BUFFER_SIZE)
        缓存区满，生产者要停; // 如何实现？
    buffer[in] = item;
    in = (in + 1) % BUFFER_SIZE;
    counter++;
}

    消费者

    while (1) {
    while(counter == 0)
        缓存区空，消费者要停; // 如何实现？
    item = buffer[out];
    out = (out + 1) % BUFFER_SIZE;
    counter--;
}

    尝试解决代码中文描述部分的问题

    我们可以使用通信的方式解决。生产者在buffer中有元素时唤醒消费者，消费者在buffer有空缺时唤醒生产者。 生产者

    while (1) {
    while(counter > BUFFER_SIZE)
        sleep();
    buffer[in] = item;
    in = (in + 1) % BUFFER_SIZE;
    counter++;
    if (counter > 0) wake_up(消费者);
}

    消费者

    while (1) {
    while(counter == 0)
        sleep();
    item = buffer[out];
    out = (out + 1) % BUFFER_SIZE;
    counter--;
    if (counter < BUFFER_SIZE) wake_up(生产者);
}

    存在的问题：两个消费者程序同时在counter == 1时执行，两个生产者同时在counter == BUFFER_SIZE - 1时执行。。。会出现错误。

    上面的示例告诉我们，要使进程有序执行，不能简单地设置一个表示buffer大小的全局变量来实现通信。

    为了避免消费者程序多于生产者（反之亦然），这个变量应该显示当前有多少个程序在等待执行。

    在编写程序时，还应该在一些位置设置与其他进程互斥的指令，防止不可控的进程切换影响程序运行顺序。

    Linux 系统使用信号量实现上述目的。

    关于信号量，可以用生产者 P 消费者 C 的示例解释： 1. 缓冲区满，P1执行，P1 sleep，sem = -1 2. P2执行，P2 sleep, sem = -2 3. C执行1次循环，wakeup P1，sem = -1 4. C再执行1次循环，wakeup P2，sem = 0 5. C再执行1次循环，sem = 1 6. P3执行，sem = 0

    sem <= 0表示当前阻塞的程序数量，sem > 0表示当前剩余资源数量 信号量是这样实现的：

    struct semaphore {
    int value; // 记录资源个数
    PCB *queue; // 记录等待在该信号量上的进程
}
P(semaphore s) { // 消费资源，荷兰语的test
    s.value--;
    if (s.value < 0) {
        sleep(s.queue);
    }
}
V(semaphore s) { // 产生资源，荷兰语的increment
    s.value++;
    if (s.value <= 0) {
        wake_up(s.queue);
    }
}

    使用信号量解决生产者消费者问题

    semaphore full = 0; // 产品到在缓冲区的数量
semaphore empty = BUFFER_SIZE; // 缓冲区剩余空间
semaphore mutex = 1; // 实现互斥信号量
void Producer(item) {
    P(empty);
    P(mutex);
    将item写入buffer;
    V(mutex);
    V(full);
}
void Consumer() {
    P(full);
    P(mutex);
    读取item，从buffer删除item;
    V(mutex);
    V(empty);
}

    临界区保护信号量

    为什么要保护信号量

    数据竞争：多个进程都在执行，不可控的进程切换可能导致共同修改信号量。

    示例：为了修改信号量empty=-1，程序 P1，P2 都会执行下面的代码：

    register = empty;                
register = register - 1;
empty = register;

    这是我们期望的执行序列，在读写变量empty时，程序 P1，P2 之间没有发生调度：

    P1.register = empty;
P1.register = P1.register - 1;
empty = P1.register;
P2.register = empty;
P2.register = P2.register - 1;
empty = P2.register;

    我们期望程序执行出的结果是-3

    然而程序的调度顺序可能是这样：

    P1.register = empty;
P1.register = P1.register - 1;
P2.register = empty;
P2.register = P2.register - 1;
empty = P1.register;
empty = P2.register;

    程序输出empty=-2，发生了读后写，写后写的问题。

    如何保护信号量

    解决竞争条件的直观方法：

    为变量“上锁”。一次只能有一个进程修改共享变量，其他修改操作必须阻塞。这段修改变量的操作序列被称为“临界区”。我们需要找到进程的临界区代码，并编写相应的进入临界区和退出临界区的代码。

    修改信号量的代码是一个临界区代码。

    临界区代码保护原则： 1. 互斥进入：一个进程在临界区执行，则其他进程不允许进入。 2. 有空让进：若干进程要求进入空闲临界区时，应尽快让一个进程进入临界区。 3. 有限等待：从进程发出请求到允许进入，不能无限等待。

    怎么使用操作系统信号量编写应用程序

    producer.c

    // 用户态程序
int main() {
    sd = sem_open("empty");
    for (int i = 1; i <= 5; ++i) {
        sem_wait(sd);
        write(fd, &i, 4);
    }
}

// 进入内核
typedef struct {
    char name[NAME_LENGTH];
    int value;
    task_struct *queue;
} semtable[20];
void sys_sem_open(char *name) {
    int sd = -1;
    if (!found(name, semtable))
        sd = semtable_create(name);
    return sd;
}
void sys_sem_wait(int sd) {
    cli(); // 关中断，防止CPU调度，只对单核有用
    if (semtable[sd].value-- < 0) {
        set_block(); // 设置自己为阻塞
        semtable_enqueue(sd); // 将自己加入semtable[sd].queue
        schedule();
    }
    sti();
}

    操作系统信号量和程序同步的实现

    从Linux0.11学点东西，这是读取磁盘块的代码：

    void bread(int dev, int block) {
    struct buffer_head *bh;
    ll_rw block(READ, bh);
    wait_on_buffer(bh);
    ...
}

    进程启动磁盘读之后睡眠，等待磁盘读完由磁盘中断将其唤醒。

    lock_buffer(buffer_head *bh) {
    cli();
    // Q : 为什么用while()? 
    // A : 唤醒好的程序在调度函数的调配下竞争运行，有利于优先级调度的实现
    while (bh->b_lock) 
        sleep_on(&bh->b_wait);
    bh->b_lock = 1;
    sti();
}

void sleep_on(struct task_struct **p) {
    struct task_struct *tmp;
    tmp = *p;
    *p = current; // 这是一个队列！
    current->state = TASK_UNINTERRUPTIBLE; // 置为阻塞态
    schedule();
    if (tmp)
        tmp->state=0;
}

    进程等待队列构建过程： 因为当前进程sleep_on()函数的局部变量会被保存在栈中，所以 tmp 的值可以在当前 PCB 中找到。找到 tmp 后，tmp 指向了下一个 PCB 的地址，所以 bh->bwait 相当于一个队列。

    磁盘读完，产生系统中断，唤醒进程

    static void read_intr() {// 磁盘中断
    ...
    end_request(1);
}
end_request(int uptodate) {
    ...
    unlock_buffer(CURRENT->bh);
}
unlock_buffer(struct buffer_head *bh) {
    bh->b_lock=0;
    /* 这个唤醒函数首先唤醒队首进程，队首进程会从sleep_on函数的schedule
     * 函数的下一句开始执行，而schedule函数后面的语句可以唤醒队列中下一个
     * 进程。这样就能不断循环，直到唤醒最后一个进程。
     */
    wake_up(&bh->b_wait);    
}

wake_up(struct task_struct **p) {
    if (p && *p) {
        (**p).state=0;
        *p=NULL;
    }
}

    死锁处理

    什么是死锁？

    把生产者和消费者的代码稍作修改，让produce先请求锁。此时buffer满了，empty = 0。这时，produce 程序停在 empty，等待 consumer 执行，但由于此时的锁 mutex 被 produce 占用，所以 consumer 也不能执行。进程双方互相等待，信号量之间发生环路等待，发生死锁。

    semaphore full = 0; // 产品到在缓冲区的数量
semaphore empty = BUFFER_SIZE; // 缓冲区剩余空间
semaphore mutex = 1; // 实现互斥信号量
void Producer(item) {
    P(mutex);
    P(empty);
    将item写入buffer;
    V(mutex);
    V(full);
}
void Consumer() {
    P(full);
    P(mutex);
    读取item，从buffer删除item;
    V(mutex);
    V(empty);
}

    死锁的成因

    1. 资源互斥使用
    2. 进程占有了一些资源，又不释放，再去申请其他资源
    3. 各自占有的资源和互相申请的资源形成了环路等待

    四个必要条件： 1. 互斥使用(Mutual exclusion)：资源的固有属性 2. 不可抢占(No preemption)：资源只能自愿放弃 3. 请求和保持(hold and wait)：进程必须占有资源，再去申请 4. 循环等待(Circular wait)：资源分配图中出现环路

    怎么处理死锁

    1. 死锁预防：破坏死锁条件
      • 1.1 在进程执行前，一次性申请所有需要的资源，不会占有资源再去申请其它资源。 缺点：需要预知未来，编程困难。许多资源分配很长时间后才使用，资源利用率低。
      • 1.2 对资源类型进行排序，资源申请必须按序进行，不会出现环路等待。 缺点： 造成资源浪费
    2. 死锁避免：判断请求会不会造成死锁
      • 如果系统中所有进程存在一个可完成的执行序列\(P_1, P_2,\dots,P_n\)，则称系统处于安全状态
      • 算法：银行家算法\(O(mn^2)\)，执行代价大。
    3. 死锁检测+恢复：发现死锁，立刻回滚进程
      • 银行家算法每次申请都要执行一次。可以考虑发现问题再处理。 缺点：回滚操作的代价大。
    4. 死锁忽略：死不死锁无所谓，我会重新启动（这是PC机操作系统使用的方法）
      • 死锁出现不是确定的，重启就可以解决，代价最小。

    银行家算法

    如果一个程序运行时的可用资源大于需要的资源，那么将可用资源分给这个程序，这个程序即可执行完毕，并释放它已经占用的资源。释放的资源重新成为可用资源。

    循环上面的过程，如果一个进程调度序列的所有程序满足上面的条件（可用资源大于需要的资源），则称这个序列是一个安全序列。

    int Available[m]; //每种资源剩余数量
int Allocation[n,m]; //已分配资源数量
int Need[n][m];//进程还需的各种资源数量
int Work[m]; //工作向量
int Finish[n]; //进程是否结束

memcpy(work, Available, sizeof(Available));
memset(Finish, 0, sizeof(Finish));
while(1){
    for(int i = 1; i <= n; ++i){
        if(Finish[i] == 0 && Need[i] <= Work){
            /* the code "Need[i]<=Work" means (all elements of 
             * Need[i]) <= (all elements of Work)
             */
            Work = Work + Allocation[i];
            // for (int j = 0; j < m; ++j) Work[j] += Allocation[j]
            Finish[i] = true;
            break;
        }
        else {
            goto end;
        }
    }
}

    if(Finish[i] == false) return deadlock;
