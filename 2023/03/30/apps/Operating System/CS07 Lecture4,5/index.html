    操作系统接口的概念

    Q1 : 操作系统为什么能够让我们更方便地使用计算机？

    A1 : 操作系统自动管理系统硬件资源，提供接口，使我们在使用程序和编写程序时无需考虑硬件的细节。

    Q2 : 上层应用如何通过操作系统使用硬件？

    A2 : 通过操作系统提供的接口。

    Q2.1 : 什么是操作系统接口？

    A2.1 : 接口负责连接两个东西，实现信号的转换，并尽量屏蔽使用的细节，例如生活中的插头，汽车的油门和方向盘等。用户使用操作系统主要有命令行，图形按钮，应用程序三种方式，这三种方式需要调用不同的接口。操作系统接口的本质是一个函数调用，所以，系统接口又称为系统调用(system_call)。

    Q2.2 : 说明使用命令行运行程序后，从调用接口的角度分析，操作系统内部发生了什么？

    A2.2 : 1. 输入命令./output "hello" 2. 执行下面的代码，通过fork()和exec()系统调用执行output程序。

    int main(int argc, char * argv[]) { 
    char cmd[20];
    while(1) {
        scanf("%s", cmd);
        if(!fork()) {exec(cmd);}
        else {wait();} 
    }
}

    Q3 : 在哪里查找操作系统的接口？

    A3 : POSIX: Portable Operating System Interface of Unix，定义了Unix系统的所有系统调用。

    系统调用的实现

    任务：用户程序调用系统函数whoami，打印出我们在系统中放置的字符串（字符串在系统引导时已经载入），去除字符串并打印。

    先实现出来——直观想法

    应用程序：

    //用户程序
main() {
    whoami(100);
}

    内核：

    //调用系统函数
whoami(int pos) {
    printf(pos, 8) // GDT[8] + pos(=100)
}
//100:"Loading system..."

    上面的代码实现了任务中的功能。但如果改变whoami的参数，那么我们将能够读取到操作系统每个内存的值！这是非常可怕的，因为内存中可能存放着各种敏感的信息，如 root 用户的密码，应用程序中的内容等。为了不让应用程序直接访问系统内核，操作系统要解决下面的问题： 1. 控制访问权限。将操作系统和应用程序隔离 2. 在控制访问权限的策略下，进入内核正确的执行系统调用

    解决第一个问题

    第一个问题其实不是由操作系统软件解决的，而是由硬件解决的。CPU的运行时状态被分为为用户态和内核态，对应的内存空间被相应分割成用户段和内核段。内核态可以任意操作内存数据，而用户态程序不能使用内核态的内存空间。我们可以使用划分等级的方式完成访问权限的控制。

    程序运行时，当前程序的请求特权级为 RPL(request privilege level)，表示给出当前选择子的那个程序的特权级别。

    当前代码所在的段的特权级被称为 CPL(current privilege level)，存放在段寄存器的最低两位（CS中的RPL就是CPL，DS中的RPL就不是了）。当前程序访问下一个段（目标段）的特权级为DPL(destination privilege level)，DPL 存储在段表 GDT 中。当程序产生访问动作时，硬件将比较当前CPL，DPL，RPL的特权级（注意特权级数值越小，特权级越高），比较策略如下： \[ \text{DPL} \geq \max \{\text{CPL}, \text{RPL}\} \] 即，DPL的特权级不能大于 CPL 和 RPL。

    解决第二个问题

    系统提供了中断int 0x80，可以把CPL，RPL的值设为0。解决第二个问题的大致思路如下图

    下面将讲述Linux0.11的具体实现方式，以printf函数的执行过程为例：

    1. 设置0x80中断
    void sched_init(void)
{ set_system_gate(0x80,&system_call); }
// 在linux/include/asm/system.h中
#define set_system_gate(n, addr) \
_set_gate(&idt[n],15,3,addr); //idt是中断向量表基址
#define _set_gate(gate_addr, type, dpl, addr)\
__asm__("movw %%dx,%%ax\n\t" "movw %0,%%dx\n\t"\
"movl %%eax,%1\n\t" "movl %%edx,%2":\
:"i"((short)(0x8000+(dpl<<13)+type<<8))),"o"(*(( \
char*)(gate_addr))),"o"(*(4+(char*)(gate_addr))),\
"d"((char*)(addr),"a"(0x00080000))
    1. 用户程序中包含一段包含int指令的代码
    // 在linux/lib/write.c中
#include <unistd.h>
_syscall3(int, write, int, fd, const char *buf, off_t, count)
// 在linux/include/unistd.h中
#define _syscall3(type,name,atype,a,btype,b,ctype,c)\
type name(atype a, btype b, ctype c) \
{   long __res;\
    __asm__ volatile("int 0x80":"=a"(__res):""(__NR_##name),
    "b"((long)(a)),"c"((long)(b)),"d"((long)(c))); 
    if(__res>=0) return (type)__res; errno=-__res;
    return -1;
}
#define __NR_write 4 //一堆连续正整数（数组下标,函数表索引）

    同时eax("a")也存放返回值，ebx("b")，ecx，edx存放3个参数

    Q4 : "=a"(__res):""(__NR_##name)是什么意思？

    A4 : 内嵌汇编的语法：__asm__(汇编语句模板: 输出部分: 输入部分: 破坏描述部分)。"a", "b", "c", "d"，表示 %eax, %ebx等寄存器。如果输入部分寄存器显示为空，表示当前操作的寄存器与上一个寄存器一样。

    1. 操作系统写中断处理，获取想调程序的编号（查 IDT 表）

      IDT表的格式和GDT表很像

      下面的代码实现了跳转到相应中断处理函数的功能：

      在linux/kernel/system_call.s中
nr_system_calls=72
.globl _system_call
_system_call: cmpl $nr_system_calls-1,%eax
ja bad_sys_call
push %ds push %es push %fs
pushl %edx pushl %ecx pushl %ebx //调用的参数
movl $0x10,%edx mov %dx,%ds mov %dx,%es //内核数据
movl $0x17,%edx mov %dx,%fs //fs可以找到用户数据
call _sys_call_table(,%eax,4) //a(,%eax,4)=a+4*eax，系统调用处理函数入口
pushl %eax //返回值压栈，留着ret_from_sys_call时用
... //其他代码
ret_from_sys_call: popl %eax, 其他pop, iret

      查找系统调用表（其中%eax = __NR_write）：

      // 在include/linux/sys.h中
fn_ptr sys_call_table[]=
 {sys_setup, sys_exit, sys_fork, sys_read, sys_write,
 ...};

      调用sys_call_table[4]函数。（在system_call.s中完成）

    总结

    int 0x80进入内核，执行system_call, 调用sys_write，向屏幕输出结果，返回用户态。
