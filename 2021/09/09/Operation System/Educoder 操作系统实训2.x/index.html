    补充： GDB设置断点的方式

    ​ 1.普通断点

    当程序执行到main.cpp文件的第5行时触发断点

    (gdb) b src/main.cpp:5
    1. 条件断点

    满足一定条件触发断点

    (gdb) b src/main.cpp:5 if cnt==10

    ​ 3.数据断点

    根据地址数据来进行设置断点，当该地址上的内容发生改变时就会触发断点。 设置数据断点有两种方式，

    一种是直接指出地址值

    (gdb) b *0x400522

    在断点处print &变量名 就能获取该变量的地址。 另一种是直接设置变量名

    (gdb) b &变量名
    1. 函数断点

    当程序执行到某个函数模块时触发断点

    (gdb) b funcName

    第二关 第一次时钟中断

    补充：GDB命令

    1. c Continue的简写，继续执行被调试程序，直至下一个断点或程序结束。

    2. b

      设置断点，上面已经介绍

    3. bt

      backtrace的简写，列出调用栈

    4. s

      相当于IDE中的step into，单步运行源代码语句

      这个命令必须在有源代码调试信息的情况下才可以使用（GCC编译时使用“-g”参数）

    5. si

      和s指令类似，单步运行汇编指令

    6. n

      相当于IDE中的step over，单步运行源代码语句

      其余同s

    7. ni

      和n指令类似，单步运行汇编指令

    第三关 第六次时钟中断

    这一关要注意，对于外部中断而言，恢复点指令是断点指令的后一条。

    loop 指令的功能是先将 ecx 寄存器减一，然后检查其值，如果其值非 0 ，则继续循环，否则中止循环，执行下一条指令。

    loop指令的上一条指令是谁？

    通过第二条info reg命令，可以观察ecx的值，为0x8440，非0，再回忆下第一条info reg，回忆ecx的值，仍为非0，得出结论，loop的上一条指令还是loop，断点指令位置仍为0x00007982

    第五关：修改版本 0 内核

    目标 ：修改版本 0 内核，使得每发生 100 次时钟中断，就在屏幕上输出一个字符‘t’和当时的进程号，如“t(0)”表示0号进程运行时发生了时钟中断。

    这里实训做的很烂。。。

    这个实训的难度就在于猜猜要改哪里。我尝试了改主函数，不行。。。改init函数，不行。pid放在代码的哪里，我也不知道。后来我看了前几关调试时命令行有一个变量叫current->state，估计这个是程序当前的pid了，make + ./run一试，还成功了。在vscode的搜索中，我搜索了一下pause相关的内容，在某个位置，出现了sys_pause这个函数，ctrl + 左键点进去，发现这个好像就是系统中断。于是在原代码基础上尝试添加以下代码。

    int sys_pause(void)
{
    current->state = TASK_INTERRUPTIBLE;
    schedule();
    
    //add following code
    if (jiffies % 100 == 0) {
            printk("t(%d)\n", current->pid);
    }
    //end
    
    return 0;
}

    然后make + ./run再再再...试，内核成功运行了，至此第五关通过。

    注意的问题：cur在前四关中都是指向1的，这关要将其指向0。至于为什么，打开config/0.11.bxrc看看。不改bochs就会报找不到硬盘的错误。

    rm -rf cur
ln -s 0 cur

    补充：

    跟踪到系统调用的陷入指令（int 0x80）执行之前

    一般而言，可以在该系统调用处设置断点，跟踪到该断点，然后使用 si 命令单步执行到陷入指令。但是，对于 main 函数里的 fork 系统调用，不能这样操作（因为 gdb 有时断点设置不够准），可以先跟踪到前一行的 move_to_user_mode 语句，然后使用 n 单步执行，即可到达 fork 系统调用的开始，此时再反汇编，找到陷入指令。

    不停运行，观察current->pid即可，答案为：

    可执行文件ls运行时（无参数），会依次执行下面这些系统调用：
1.(time)(ioctl)(brk)(brk)(brk)
2.(brk)(brk)(open)(fstat)(brk)
3.(read)(read)(close)(ioctl)(write)
4.(write)(close)(exit)()()

    ls在6号进程中执行，后面由 6 号进程引发的系统调用才是ls 执行的系统调用，所以只需要记录current->pid = 6时的所有系统函数调用！

    第三关：版本1内核执行的完整系统调用序列

    补充：GDB断点的启用和禁用

    • disable 1 ：禁用第一个断点；
    • enable 2 ：启用第二个断点；
    • enable once 1 ：下次用了第一个断点，第一个断点就被禁用；
    # 2号断点为无条件的system_call
b system_call
commands 2
p $eax
p sys_call_table[$eax]
end
disable 2
# 3号断点为有条件的system_call
b system_call if $eax!=29 || current->pid!=0
commands 3
p $eax
p sys_call_table[$eax]
end
c

# 接下来要不停的开关断点，在出现0号进程pause调用时，把它们跳过去
disable 2
enable 3
c
# 如果本次显示的断点没有发生0号进程pause调用，不用跳过，执行一次system_call
disable 3
enable 2

    第四关：在版本1中输出系统调用信息

    修改system_call.asm中的system_call:函数

    system_call:
    cmpl $nr_system_calls-1,%eax
    ja bad_sys_call
    push %ds
    push %es
    push %fs
    pushl %edx
    pushl %ecx      # push %ebx,%ecx,%edx as parameters
    pushl %ebx      # to the system call
    movl $0x10,%edx     # set up ds,es to kernel space
    mov %dx,%ds
    mov %dx,%es
    movl $0x17,%edx     # fs points to local data space
    mov %dx,%fs
    pushl %eax #新加入的指令1
    call myfunc #新加入的指令2
    popl %eax #新加入的指令3
    call sys_call_table(,%eax,4)
    pushl %eax
    movl current,%eax
    cmpl $0,state(%eax)     # state
    jne reschedule
    cmpl $0,counter(%eax)       # counter
    je reschedule

    修改fork.c，在其中添加

    int baghdnkdn = -1;//全局变量，随机取名，不要与已有的变量名重复
void myfunc (int eax) {
    if (baghdnkdn == current -> pid && eax == 29 && baghdnkdn == 0) return;
    else {
        printk("%d(%d)", eax, current->pid);
    }
    baghdnkdn = current -> pid;
}
